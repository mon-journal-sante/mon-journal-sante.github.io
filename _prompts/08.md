## Bug #1 — "Ajouter" bouton dans le formulaire Repas

### Current behavior

The "Ajouter" button in the meal add form (`RepasForm` inside `AujourdhuiScreen.tsx`) is disabled as long as `aliments` is empty. The user cannot submit a meal without at least one aliment tag.

### Expected behavior

The user may describe their meal in the "Note" textarea instead of using aliment tags. The "Ajouter" button must be enabled when **either** `aliments.length > 0` **or** `note.trim()` is non-empty. It stays disabled only when both are empty.

### Change

In `src/screens/AujourdhuiScreen.tsx`, inside `RepasForm`, update the "Ajouter" button:

```tsx
// before
disabled={aliments.length === 0}
style={{ backgroundColor: aliments.length === 0 ? 'var(--color-text-muted)' : 'var(--color-primary)' }}

// after
disabled={aliments.length === 0 && !note.trim()}
style={{ backgroundColor: aliments.length === 0 && !note.trim() ? 'var(--color-text-muted)' : 'var(--color-primary)' }}
```

  - Opening the meal add form: "Ajouter" is disabled when both aliments and note are empty.

  - Adding at least one aliment tag enables "Ajouter" even with an empty note.
  - "Ajouter" stays disabled when both the aliments list is empty and the note is blank.
  - Saving a meal with only a note (no aliment tags) persists correctly and appears in Historique.

## Feature #2 — Champ Température dans la section Symptômes

### Need

The user takes their temperature and wants to log it for future correlations (e.g. migraine days vs. elevated temperature).

### Data schema — `src/db.ts`

Add one optional field to `JournalEntry`:

```ts
temperature?: number  // in °C, e.g. 37.5
```

### UI — `src/screens/AujourdhuiScreen.tsx`

Inside the Symptômes section, add a numeric input for temperature. Place it as the first field of the section, before the migraine toggle.

- Label: `"Température (°C)"`
- Input: `type="number"`, `step="0.1"`, `min="35"`, `max="42"`, `placeholder="37.0"`
- Optional — leave empty if the user has not taken their temperature.
- On change: update the `temperature` field in local state (parse as float, store `undefined` if the field is empty or not a valid number).
- Persist to IndexedDB alongside the other Symptômes fields on "Enregistrer la journée".
- On load: restore from the saved entry like the other fields.

### Historique — `src/screens/HistoriqueScreen.tsx`

In the detail view, inside the **Card: Symptômes**, add one line when `entry.temperature` is defined:

```
🌡️ Température : 37.5 °C
```

Include this line regardless of whether `entry.migraine` is set (temperature is independent).
Update the card's visibility condition to also show when `entry.temperature !== undefined`.

### Success criteria

- `npm run dev`:
  - The temperature input appears at the top of the Symptômes section.
  - Entering a value and saving persists it to IndexedDB.
  - Reloading the page restores the temperature value in the input.
  - Leaving the field empty saves no temperature (field stays `undefined`).
  - The Historique detail view shows the temperature line when a value was saved.
  - The Symptômes card in Historique is visible even if only `temperature` is set (no migraine data).
- All other sections and screens are unchanged.

## Feature #3 — Modifier une entrée depuis l'Historique

### Need

From the detail view of a past entry, the user wants to edit any field and save the changes back to IndexedDB.

### Approach

Reuse `AujourdhuiScreen` by giving it optional props instead of duplicating the form logic.

---

### 1. Update `src/screens/AujourdhuiScreen.tsx`

Change the component signature to accept optional props:

```ts
export function AujourdhuiScreen({ date: dateProp, onSaved }: { date?: string; onSaved?: () => void } = {})
```

- Replace `const today = todayISO()` with `const today = dateProp ?? todayISO()`.
- **Title block**: when `onSaved` is defined (edit mode), replace the "Aujourd'hui" heading with `"Modifier"` and keep the formatted date line unchanged.
- **Save button**: label is `"Enregistrer les modifications"` when `onSaved` is defined, `"Enregistrer la journée"` otherwise.
- **After save**: always show the toast. Additionally, when `onSaved` is defined, call `onSaved()` after the same 2500 ms delay (when the toast disappears).

No other logic changes.

---

### 2. Update `src/screens/HistoriqueScreen.tsx`

Add an `editing` boolean state to `HistoriqueScreen` (alongside the existing `entries` and `selected` states):

```ts
const [editing, setEditing] = useState(false)
```

**When `editing === true`** (and `selectedEntry` exists): render `AujourdhuiScreen` with:
- `date={selectedEntry.date}`
- `onSaved` callback that: calls `getAllEntries().then(setEntries)` to refresh the list, then calls `setEditing(false)` — the user lands back on the (now up-to-date) detail view.

**In the detail view header** (`DetailView` component): add a `onEdit` prop and render a `"Modifier"` button on the right side (replacing the invisible placeholder span). Style it like the back button — text `var(--color-primary)`, transparent bg, no border.

Wire up: in `HistoriqueScreen`, pass `onEdit={() => setEditing(true)}` to `DetailView`.

---

### Success criteria

- `npm run build` completes without errors, no TypeScript errors.
- `npm run dev`:
  - The "Aujourd'hui" tab behaves exactly as before (no visible change).
  - In Historique, opening a detail view shows a "Modifier" button in the header.
  - Tapping "Modifier" opens the full edit form pre-filled with the entry's data.
  - Editing any field and tapping "Enregistrer les modifications" shows the toast, then returns to the detail view with updated data.
  - The Historique list also reflects the updated data.
  - "← Retour" from the edit form is not needed — `onSaved` handles navigation back.

### What NOT to do

- Only change `src/screens/AujourdhuiScreen.tsx` and `src/screens/HistoriqueScreen.tsx`.
- No new npm dependencies.
- No test files.
- Do not extract shared components or utilities.
